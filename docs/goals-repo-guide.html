Understand the Goals repo

This guide condenses the teaching notes for the Goals repository into one standalone reference. It explains how the repo is built, how the workflow functions, which files matter most, and how to use or modify the project without getting lost.

1. The Core Mental Model

Goals is a file-backed workflow layer for native agent goal loops. Claude and Codex keep attention on the work through /goal. Goals keeps durable state, phases, proof, review gates, decisions, worktrees, checkpoints, dashboard views, and safety checks.

One-line architecture: Goals stores a durable event history, derives current state, gives Claude or Codex the next phase prompt, then checks proof before letting the goal advance.
events.jsonlPermanent history
goal.jsonCurrent summary
goals runNext native /goal handoff
agent worksCodex or Claude executes
evidenceProof is recorded
checkpointCurrent validation is clear
reviewGate checks proof
acceptMove to next phase
events -> snapshot -> agent prompt -> work -> evidence -> checkpoint -> review -> accept -> next prompt

The easiest analogy is a bank account: 

events.jsonl    = transaction history
goal.json       = current balance
dashboard.html  = friendly banking screen

2. Day-to-Day Use

The main workflow is small. Create a goal, move into the generated worktree, ask Goals for the next agent handoff, then record and review proof as phases finish. 

uv run goals create "Add tags to tasks and update tests" --adapter codex
cd ../your-repo-add-tags-to-tasks-and-update-tests
uv run goals run --adapter codex

After an agent finishes a phase:

uv run goals phase evidence P1 --file .agent-workflow/goals/<goal-id>/evidence-p1.json
uv run goals checkpoint current
uv run goals phase review P1
uv run goals phase accept P1
uv run goals run --adapter codex
goals brief Human-friendly status: what is happening, what needs the user, and what the agent can do next.
goals checkpoint current Plain-language validation point: what is being checked, who is waiting, what proof exists, and the next safe step.
goals issues Coordinator/debug report: blockers, incomplete checkpoints, missing proof, failed gates, unresolved claims, and user-worthy decisions.
goals run --adapter codex Prints the next native Codex /goal handoff. It does not launch or control Codex.

3. State: events.jsonl, goal.json, dashboard.html

The repo uses an event-sourced shape. Goals records what happened first, then derives current state and views from that history.

Artifact Meaning Trust level What to do if wrong
events.jsonl The permanent log of goal events. Source of truth. Inspect events carefully. If invalid, state cannot be safely derived.
goal.json The current snapshot derived from events. Useful summary, not the source of truth. Run uv run goals repair.
dashboard.html Read-only human explanation of goal state. Presentation layer. Run uv run goals dashboard.

The implementation lives mostly in src/goals/storage.py. EventStore.append appends the event, rewrites the event log atomically, derives the snapshot, and writes goal.json. 

events.jsonl = what happened
goal.json    = where we are now
dashboard    = how to explain it to people

4. The Phase Loop

A phase is one step inside a goal. The default phases are:

  1. Confirm outcome and plan
  2. Inspect project and choose approach
  3. Execute the core change
  4. Review, explain, and close
pendingPhase exists
in progressAgent starts work
evidenceProof is recorded
reviewGate checks proof
acceptedNext phase allowed

The three commands are separate because they do different jobs: 

phase evidence = what the agent claims it did
phase review   = whether the proof is good enough
phase accept   = permission to move forward

The review gate is deliberately simple. A phase fails review if there is no evidence, unmet acceptance criteria, ambiguity, confidence below 0.7, or no checks/tests recorded. After repeated failed attempts, review becomes blocked so the agent does not loop forever.

If you understand these files, you understand the spine of the repo.

6. Decisions and User Interruptions

Goals should reduce unnecessary user interruptions while still surfacing meaningful risk. The core rule is in decisions.py. 

blocking decision              -> ask user
any high-risk option           -> ask user
medium-risk and irreversible   -> ask user
low-risk or reversible choice  -> agent decides and records assumption
Practical diagnosis: If the agent asks too many small questions, inspect decisions.py. If the issue report creates too many user questions, inspect issues.py. If the wording is bad, inspect brief.py and dashboard.py.

Tool approval is separate from product/technical decisions. External services, connectors, destructive actions, production-impacting actions, and costly actions are governed through registries/permissions.yml and applied by permission_policy.py.

7. Checkpoints and Safe Continuation

Checkpoints are phase-level stop signs. They make validation explicit before an agent reviews or accepts a phase. 

uv run goals checkpoint record P1 CP-plan \
  --title "Confirm the plan" \
  --kind human_validation \
  --status needs_user \
  --needs-user
uv run goals checkpoint current
uv run goals checkpoint waive P1 CP-plan --reason "User confirmed the plan."
Rule: Required checkpoints must pass or be waived before phase review and acceptance can move forward. Optional checkpoints can inform the dashboard without blocking the agent.

Use checkpoints for human validation, approvals, understanding checks, external reviews, handoffs, and custom validation points. They are generic on purpose: a coding project, research brief, creative review, operations process, or high-stakes support goal can all use the same checkpoint shape.

8. Worktrees

A worktree is the physical workspace for a goal. It gives one goal its own folder and branch so changes are easier to attribute and review. 

base repo:       my-app
goal id:         add-tags-to-tasks
branch:          goal/add-tags-to-tasks
worktree folder: my-app-add-tags-to-tasks

Goals records this as a WorktreeLease: base repo, base branch, worktree path, branch, creation time, and topology mode.

Why dirty repos are refused: Goals needs a clean starting line. If the repo already has uncommitted changes, the agent cannot reliably know what existed before the goal and what the goal itself changed.

In v1 the default topology is one goal, one worktree. The data model is ready for parallel, swarm, and rewrite modes, while deeper multi-worktree orchestration is future work.

9. Registries, Skills, and Plugins

Goals keeps portable ecosystem configuration in YAML registries. This lets the project recommend tools without hardcoding one person's local setup.

skills.yml Skill routing hints such as decision explainer, architecture map, testing, security review, and docs update.
plugins.yml Plugin routing hints such as GitHub, Browser, Build Web Apps, and Product Design.
permissions.yml Human-readable policy for agent-decide, ask-user, or deny decisions.
gates.yml Gate names and scanner groups. 
uv run goals ecosystem recommend
uv run goals ecosystem discover
uv run goals ecosystem sync
uv run goals ecosystem audit

ecosystem discover inspects local skills/plugins/adapters. ecosystem sync proposes portable registry additions. registry.py rejects unknown critical fields so unsafe YAML does not silently become trusted behavior.

10. Dashboard, Brief, and Architecture Map

The dashboard is a read-only explanation layer. It should not mutate goal state. State changes happen through CLI commands that append events.

For non-technical users

  • goals brief explains what is happening.
  • goals checkpoint current explains the current validation point.
  • goals dashboard renders a status page.
  • goals decision brief shows only important decisions.

For technical reviewers

  • goals architecture show writes architecture.md.
  • goals architecture brief summarizes map quality.
  • goals architecture check compares changed files with the map.

The product idea is the same state, different explanations for different audiences. 

goal.json -> brief -> plain-language status
goal.json -> dashboard.html -> read-only visual status
goal.json -> architecture.md -> technical shape of the goal
goal.json -> issues -> coordinator/debug report

11. Validation, Safety, and Debugging

goals validate and goals safety-check sound similar, but they protect different things.

Command Question it answers Typical fix
uv run goals validate Is Goals state structurally trustworthy? Run uv run goals repair if snapshot differs from events.
uv run goals safety-check --mode local Is local work free of obvious safety findings while allowing generated state? Fix reported scanner findings.
uv run goals safety-check --mode publish Is this safe to publish or share? Remove private generated state, secrets, local paths, unsafe text, or license gaps.

Debug map

goal.json stale/wrong       -> goals repair
registry YAML broken        -> goals validate
agent asks too often        -> decisions.py + issues.py
phase cannot advance        -> gates.py + phase evidence
dashboard looks wrong       -> dashboard.py, then goals dashboard
Codex prompt looks wrong    -> mode_a.py
worktree/branch issue       -> git_ops.py + runtime.py
merge/migration risk        -> goals merge-check
public release concern      -> goals safety-check --mode publish

12. How to Safely Modify the Repo

Most feature changes follow this path: 

model -> event/storage -> runtime/analysis -> CLI -> read model -> tests

Example: adding a new kind of tracked evidence.

  1. Inspect models.py to understand the data shape.
  2. Add or reuse an EventType.
  3. Teach storage.py how to replay the event into goal.json.
  4. Add runtime or analysis behavior if needed.
  5. Add a CLI command in cli.py if users or agents need one.
  6. Show it in brief, dashboard, issues, or mode_a if relevant.
  7. Add focused tests.
Rule of thumb: If you are changing what Goals knows, start with models.py. If you are changing how Goals explains what it knows, start with brief.py or dashboard.py. If you are changing what agents are told to do, start with mode_a.py.

14. Mode A and Mode B

The current repo is Mode A first. 

Mode A:
Goals prepares the work.
Codex or Claude does the work.
Goals records proof and checks progress.

goals run --adapter codex prints a Codex /goal handoff. It does not launch Codex, control Codex, or replace the native goal loop.

Mode B is future work: a standalone runner shape for local AI or schedulers. Runtime interfaces are present, but the standalone runner is intentionally not complete in this version.

15. Self-Check Quiz

If you can answer these, you understand the practical 80 percent of the repo.

  1. What is the source of truth: events.jsonl, goal.json, or dashboard.html?
  2. What does goals run --adapter codex actually do?
  3. Why does a phase need evidence, review, and accept?
  4. Where do skill/plugin recommendations come from?
  5. What kind of choices should reach the user?
  6. What does goals checkpoint current tell you?
  7. Why does goals create refuse a dirty repo?
  8. What is the difference between goals validate and goals safety-check?
  9. If the dashboard UI needs improvement, which file do you inspect first?

Answer key

  1. events.jsonl is the source of truth.
  2. It prints the next native Codex /goal handoff.
  3. Evidence records proof, review checks proof, accept moves to the next phase.
  4. YAML registries plus local discovery, scored by ecosystem code.
  5. Blocking, high-risk, irreversible, external, destructive, costly, or direction-changing choices.
  6. The current validation point, who is waiting, what proof exists, and the next safe step.
  7. Because dirty starting state makes later diffs and proof ambiguous.
  8. validate checks internal state integrity; safety-check checks public-sharing safety.
  9. src/goals/dashboard.py.
